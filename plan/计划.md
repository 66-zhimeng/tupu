### 1. 系统核心架构

- **展现形式**：基于力导向图（Force-Directed Graph）的二维拓扑网络可视化。
- **技术栈选型**：
- 前端：WebGL/Canvas 图渲染引擎（推荐 AntV G6），支持百万级节点和局部物理引擎。
- 后端：Python API（如 FastAPI/Django）。
- 数据库：PostgreSQL（存储属性与计算逻辑）+ Neo4j（存储拓扑网络）+ MinIO/OSS（存储物理文件）。

### 2. 任务模型与数据计算规则

- **N级嵌套结构**：支持子任务的无限层级划分（树状结构）。
- **叶子节点录入原则**：仅允许在最底层的“叶子节点”手动录入“预估工时（小时）”与“完成进度（0-100%）”。
- **递归汇总（自底向上）**：
- **体积（工时）计算**：父节点工时 = 所有直接子节点工时之和。
- **进度（加权）计算**：父节点/里程碑进度 = [ ∑ (子节点工时 × 子节点进度百分比) ] / 当前节点总工时。
- 使用 PostgreSQL 的 `WITH RECURSIVE` 语法实现层级数据的自动汇总。

### 3. 图谱视觉与渲染规则

- **节点大小**：节点半径（Radius）严格映射该节点的“总工时”。
- **节点进度**：采用饼图/扇形填充机制呈现进度 。完成部分按负责人专属颜色填充（多人负责则按工时比例分块着色），未完成部分留空或置灰。
- **边缘连线**：采用**带箭头的原生直线**表示前后置依赖。默认降低全局连线透明度，悬停/选中节点时高亮相关连线。
- **布局与聚集**：取消全局 Y 轴引力。里程碑作为隐形的聚类中心（Cluster Center），归属同一里程碑的任务在其周围聚集。
- **展开与缩放**：点击父节点可原地展开其内部子任务网络，物理引擎自动增加局部斥力边界，将周围其他节点推开以避免重叠。

### 4. 依赖拓扑与冲突处理

- **跨层级连线**：允许不同层级、不同父任务下的子节点之间直接建立依赖关系。
- **死循环检测**：在 Neo4j 写入新依赖时，检测是否存在闭环（A 依赖 B，B 最终依赖 A）。
- **前端红线阻断**：若检测到死循环，不强制拦截数据库写入，但将该边标记为冲突（`is_conflict = true`），前端强制渲染为高亮红色粗线，提示人工排查。

### 5. AI 增量分析引擎

- **局部连线推理**：不进行全局 GraphRAG 重算。新增任务时，仅提取同属一个父级或里程碑的 1-2 度局部拓扑子图。
- **增量执行**：将新任务文本与局部子图数据交由大语言模型（LLM）推理，输出前后置关系建议，并直接转换为数据库连线指令。

### 6. 面板互动与文件关联

- **通用交互面板**：点击任意层级节点（里程碑、父任务、子任务），弹出侧边抽屉。支持修改状态、增删依赖线。
- **多态文件存储**：采用实体多态关联（`entity_id` + `entity_type`），将各层级节点与底层的物理附件相绑定。仅提供纯粹的文件上传、下载、删除功能，**不包含**文件解析与 AI 内容对话。

### 1. 权限与角色控制 (RBAC)

- **细粒度权限**：
- **里程碑/全局级**：仅项目管理员或指定负责人可创建、修改里程碑，以及调整跨里程碑的任务归属。
- **任务级**：任务的“预估工时”和“进度”仅允许该任务的**设定负责人**或**上级父任务负责人**进行修改。
- **拓扑级**：任何人均可调用 AI 建议连线，但写入数据库前需经相关任务负责人确认。

### 2. 操作审计与历史日志 (Audit Log)

- **变更追溯**：在 PostgreSQL 中增加独立的日志表。记录所有关键操作，包括：工时修改（原值与新值）、进度更新、文件增删、依赖连线的建立与解除。
- **面板展示**：在每个任务节点的交互抽屉面板中，增加“操作日志”模块，按时间轴倒序展示该节点的所有变更历史及操作人。

### 3. 并发控制与数据一致性

- **乐观锁机制**：在 PostgreSQL 的任务表中引入版本号（`version`）字段。当多名用户同时打开同一个任务面板并尝试保存修改（如同时更新进度）时，系统通过版本号校验，防止后提交的数据覆盖先提交的数据，并向用户报错提示重新拉取。

### 4. 冲突告警与消息推送

- **主动通知**：当系统（或 AI）在 Neo4j 中写入依赖关系并检测出死循环（触发 `is_conflict = true`）时，除了前端渲染红色高光连线外，后端需触发异步消息机制，通过站内信或 Webhook（如钉钉/飞书/企业微信接口）向冲突链路上的所有任务负责人发送告警通知。

### 5. 前端渲染性能降级策略

- **阈值设定**：明确 WebGL 引擎的单屏渲染节点上限（例如 2000 个节点）。
- **自动折叠**：当局部展开导致节点总数接近性能阈值时，系统自动执行降级策略：强制收起/折叠所有进度已达到 100% 的父任务节点，将其内部的子网络隐藏，仅保留父节点本身的 100% 扇形图 。

### 6. 多维检索与视图过滤 (Search & Filtering)

- **全局检索**：在前端顶部提供搜索框。支持通过任务 ID、标题关键字或负责人进行精确检索。
- **视觉聚焦定位**：检索命中特定任务后，前端图谱自动将该节点平移至屏幕中央并放大，同时将全图其他无关节点的透明度降低至 10%，并高亮显示该节点的所有前后置连线。
- **动态过滤**：支持按条件（如：仅显示我的任务、仅显示进度 < 100% 的任务、仅显示包含冲突的任务）过滤图谱。被过滤掉的节点在视图中隐藏，物理引擎自动重新计算剩余节点的布局。

### 7. 绝对时间属性与逾期预警 (Time Management)

- **时间字段**：除了“预估工时（小时）”，任务和里程碑强制增加 `start_date`（计划开始日期）和 `due_date`（计划完成日期）两个绝对时间字段。
- **逾期视觉标识**：当系统当前时间超过任务的 `due_date`，且该任务的完成进度 < 100% 时，前端为该节点添加警示标识（例如：节点外圈渲染红色的虚线边框）。该标识需在视觉上与依赖冲突的“红色粗连线”明确区分。

### 8. 图谱快照与历史比对 (Graph Snapshots)

- **定时快照生成**：系统支持按周期（如每周五晚）或通过手动触发，将当前整个项目图谱的节点状态、进度数据及连线拓扑结构以 JSON 格式完整持久化存储到 PostgreSQL 中。
- **版本回退与只读查阅**：用户可选择查看任意历史快照。在快照模式下，图谱为只读状态，物理引擎停止动态计算，完全复现当时的静态节点位置与完成度，用于项目复盘。

### 9. 数据初始化与批量导入导出 (Data I/O)

- **模板导入**：支持通过标准 Excel/CSV 模板批量导入初始任务层级与工时数据。后端接收数据后，自动生成基础的从属关系（`BELONGS_TO`）并写入 Neo4j。
- **结构化导出**：支持将当前的图谱拓扑数据导出为结构化的 Excel 报表（包含任务树状层级、依赖任务列表、当前进度统计），用于脱离系统的线下汇报。

### 10. AI 接口限流与成本控制 (AI API Control)

- **调用控制**：针对“AI 分析任务关系并连线”功能，在后端 API 层设置限流阈值（Rate Limiting）。例如限制单用户/单项目的每日最大请求次数或 Token 消耗总量。
- **人工熔断**：如果 AI 返回的局部拓扑连线建议连续出现逻辑错误或极低的置信度，系统允许管理员在后台一键关闭该项目的 AI 辅助连线功能，强制切换为纯人工连线模式。
